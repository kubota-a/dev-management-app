## 📌今後のリファクタリング方針

本アプリは、提出物として制限時間内に申請・承認・進捗管理・予算管理の一連の業務フローを成立させることを最優先に実装しました。今後はポートフォリオとして、提出時に見えた設計上の課題を整理し、以下の方針で段階的に改善していく予定です。

### 1. DB設計の見直し

提出版では、制限時間内で主要フローを完成させるため、一部のデータ構造を簡略化しています。

特に、タスク担当者を名前テキストで管理しているため、表記ゆれや同姓同名への対応、担当者別の正確な負荷集計に課題があります。また、ユーザーは基本的に単一部門所属として扱っていますが、実運用では複数部門を兼務するケースも想定されます。

今後は、以下のような改善を検討しています。

* タスク担当者をテキストではなくIDで管理する
* 部門メンバー管理テーブルを追加する
* ユーザーが複数部門に所属できる構造にする
* 月次報告を最新1件ではなく履歴として管理する
* 論理削除や監査ログを追加し、業務履歴を追いやすくする

### 2. UI整合性の改善

提出版では、各画面の使いやすさを優先して実装した結果、テンプレートごとの直書きCSSやJSが残っています。今後は、画面ごとの見た目や操作感のばらつきを減らし、ポートフォリオとして見やすい状態に整えていきます。

主な改善方針は以下です。

* 直書きCSS / JSを整理する
* 共通コンポーネントを増やす
* カード、ボタン、ラベル、余白、見出しのルールを統一する
* 案件バナーや承認ステッパーなどを共通化する
* 申請者・部門管理者・本部管理者で操作感を揃える
* PC表示とスマホ表示の情報優先度を再整理する

### 3. 案件進捗管理画面のUX強化

案件進捗管理画面は、本アプリの中でも特に力を入れて実装した画面です。進捗報告・予算実績入力・月次報告が分断されている課題に対し、タスク進捗、予算実績、月次報告を1画面に集約し、現場担当者の入力負担を減らすことを目指しました。

今後は、この画面をさらに使いやすくするため、以下を検討しています。

* 「更新する」「変更を戻す」の操作導線を改善する
* 操作単位で取り消せるUndo機能を追加する
* 進捗更新の一時保存に対応する
* 月次報告を履歴管理できるようにする
* 予算実績入力時のシミュレーション表示を改善する
* スマホでのスライダー操作やタップ操作の誤操作対策を強化する

### 4. 一覧性・履歴閲覧の改善

提出版では、各ロールのトップ画面に案件やタスクの情報を多めに表示しています。PoCとしては主要な情報を1画面で確認できることを優先しましたが、実運用では案件数やタスク数が増えるため、一覧性と検索性の改善が必要です。

今後は以下のように、トップ画面と専用一覧画面の役割を分けていく予定です。

* トップ画面には重要度の高い情報のみ表示する
* 全件確認用の専用一覧画面を追加する
* 検索・絞り込み・ステータス別表示に対応する
* 必要に応じてサーバー側ページングを導入する
* 完了済み案件は履歴閲覧画面に分離する
* 進捗履歴、予算実績履歴、完了日、最終報告を確認できるようにする

### 5. 通知・業務フロー拡張は将来案として整理

提出版では、申請・承認・却下・完了認定など、主要な業務フローに関わる通知を中心に実装しました。今後は、通知一覧や再申請、申請取消、差し戻し、条件付き承認なども検討できますが、これらは状態遷移や権限制御が複雑になるため、まずはDB設計とUI整合性の改善を優先します。

将来的な拡張候補は以下です。

* 通知一覧画面の追加
* 遅延・予算注意などの通知追加
* 申請取消機能
* 却下後の修正・再申請機能
* 差し戻し機能
* 条件付き承認・承認予算額の調整
* 承認コメント・最終決裁コメントの追加

## 📌優先順位

今後の作業では、まずデータ構造の整理を行い、その上でUI/UXと一覧性を段階的に改善していく予定です。

1. DB設計の見直し
2. 案件進捗管理画面のUX強化
3. UI整合性の改善
4. 一覧性・履歴閲覧の改善
5. 通知・業務フロー拡張

特に、まずはポートフォリオとして見たときの完成度を高めるため、画面の統一感と、最もこだわって実装した案件進捗管理画面の使いやすさを優先して改善していきます。
